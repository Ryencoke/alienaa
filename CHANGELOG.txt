BUGFIX — DAEMON.PHP SPIN/ROLL RESULT-BEFORE-REVEAL (6th report, new root
cause found — the pre-submit gap)
------------------------------------------------------
Five prior fixes (BATCH-61 through BATCH-66) each fixed a real bug in the
POST-response reveal sequence, but the user kept seeing the same symptom.
Found a SEPARATE, previously-unaddressed gap in the PRE-submit phase: the
click handlers on the Spin/Roll buttons only ever added a CSS blur class
("rolling") to the EXISTING reel/dice elements — they never touched
innerHTML or the reel-win/reel-lose/win/lose class already applied from
the previous play. So the moment you clicked Spin or Roll, the PREVIOUS
result's actual symbols and win/lose glow color sat there, blurred but
still legible, for the entire network round-trip (however long that
takes) before the server even responds and the already-fixed hidden-then-
revealed sequence takes over. From the player's view that's two back-to-
back "reveals" — the stale blurred one right on click, then the real one
— which reads exactly like "it shows the result before the reels finish."
Fixed by having both pre-submit handlers reset content AND class to a
genuinely random neutral state immediately, synchronously, the instant
Spin/Roll is clicked — never leaving any real result on screen at any
point between click and the deliberate reveal. Hoisted the symbol/face
arrays (SLOT_SYMS/DICE_FACES) to a shared scope so the pre-submit reset
and the post-response reveal can't drift out of sync with each other.
Not independently verified against a live browser (none available here)
-- flagged, not declared certain, given the history on this one.

BUGFIX — NOTIFICATIONS REAPPEARING AFTER DISMISS (home.php, lib.php)
------------------------------------------------------
Root cause: `seed_player_notifications()` relied entirely on INSERT IGNORE
plus a uq_pn_ref unique key added via a best-effort ALTER TABLE wrapped in
a silent try/catch in `ensure_player_notifications_table()`. If that ALTER
ever failed against pre-existing duplicate data, dedup provided nothing, so
every ~12s poll could insert a fresh duplicate for the same still-unread
message/transfer -- dismissed notifications appeared to "reappear" and the
unread count didn't go down. Fixed with explicit SELECT-before-INSERT
dedup at the application level, independent of DB constraint state. Also
fixed notification text overlapping other text (missing
overflow-wrap/word-break on the notification body containers, both server
render and the JS live-poll render() path).

NEW FEATURE — BATTLE-XP TITHE & GUILD LEVELING
------------------------------------------------------
Players can now pledge a share (0–100%) of the XP they earn in combat to
their Syndicate, and Syndicates now actually LEVEL UP from accrued XP.

Background: syndicates already had xp/level columns, but nothing ever
raised the level — XP just accumulated and level sat at 1 forever. This
adds the real leveling mechanic and a player-driven way to feed it.

[lib.php] New shared helpers:
- syn_xp_for_level(L): cumulative XP to reach a guild level
